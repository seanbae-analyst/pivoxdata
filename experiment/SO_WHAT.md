# The "so what" experiment — what actually happens when an LLM analyzes your dirty CSV

> **Claim under test:** an LLM asked to analyze a CSV does not *error* on dirty data — it
> returns confident numbers computed on top of the defects. Clean the file first and the
> same questions come back correct, direct, and reproducible.
>
> Everything below is generated by [`run.mjs`](run.mjs) + the Claude CLI; the raw
> transcripts are committed under [`results/`](results/) unedited. Nothing is retyped.

## Protocol

- **Dirty file:** [`dirty.csv`](dirty.csv) — the shipped `sales_pipeline.csv` demo sample,
  untouched. 44 rows: 2 exact duplicate rows, 7 placeholder dates (`1900-01-01`), 8 sentinel
  amounts (`unknown`/`TBD`), mixed date formats, `apac`/`Apac` + `GLOBEX`/`Globex` spelling
  variants, 44 real emails, 32% missing region. DataReady score: **74**.
- **Clean file:** [`clean.csv`](clean.csv) — the same file after DataReady's recommended
  one-click set (dedupe, ISO dates, clear placeholders & sentinels, drop empty column, trim,
  unify variants to the most common form, mask PII). Missing values stay missing — the tool
  refuses to impute. Score: **95**.
- **Identical prompt** ([`prompt_dirty.txt`](prompt_dirty.txt) / [`prompt_clean.txt`](prompt_clean.txt)):
  five analyst questions, "use ONLY the CSV."
- **Models:** `claude-sonnet-5` (frontier) and `claude-haiku-4-5` (the cheap tier real
  pipelines actually batch with). Single-shot runs via Claude CLI; sonnet run twice on the
  dirty file to test reproducibility.
- **Ground truth:** computed deterministically by [`run.mjs`](run.mjs) →
  [`ground_truth.json`](ground_truth.json).

## The honest findings — including the one that surprised us

**1. The cheap tier gets confidently wrong numbers.** Haiku reported a pipeline total
inflated by **$18,500** and named **1900-01-01** the earliest close date — while *noticing*
both defects in its caveats. The wrong number is the headline; the doubt is a footnote.
That's worse than an error: an error stops the report, a confident-but-wrong total ships.

**2. The frontier model self-cleans — by improvising un-audited judgments.** Sonnet got the
numbers right on the dirty file, and to do it, it silently made the *exact* judgment calls
DataReady makes explicit: dedupe D-1004, exclude the sentinels, discount 1900-01-01, merge
`GLOBEX`/`Globex`. Nobody reviewed those calls; they're remade from scratch inside every
prompt. And they wobble: **the same model on the same file gave opposite headline answers
for "earliest date" in two consecutive runs.** In-context cleaning is analytics that changes
between runs.

**3. On the clean file, every answer is direct, correct, and boring** — because the
judgments were made **once**, by a human, explicitly, in the cleaning step, instead of
per-query by a model. Boring is what you want your pipeline totals to be.

**4. Smartness can't undo exposure.** 44 real employee emails entered the prompt in every
dirty run — before any model, however careful, could "handle" them. The clean file sent 0.
PII cleaning after upload is too late by definition.

**5. This was the easy mode.** 44 rows fit in one prompt, so a frontier model *could*
hand-audit every cell. The anchor dataset this tool is validated on is 295,810 rows — it
does not fit in a context window, and nobody hand-audits it in-context. Self-cleaning is a
small-file luxury; a deterministic pass in front of the model is what scales.

## One sentence

**LLMs don't fail on dirty data — they either ship a wrong number with a confident face
(cheap tier), or quietly re-make your cleaning decisions differently every run (frontier
tier). Cleaning once, explicitly, before the prompt is the only version that is correct,
reproducible, and doesn't leak PII.**
